|---|---|---|

```

```

## A. 技术风险

### A1. 双 Agent 工程复杂度 🔴

**触发条件**：
- CTA 与 LCA 接口不稳定
- 互校循环引入额外延迟（>2 秒）
- 双 Agent 状态同步失败

**影响评估**：
- MVP 延期 2-4 周（占位文件提到）
- 系统稳定性下降（崩溃率 > 5%）
- LLM API 成本翻倍（互校需要多次 LLM 调用）

**缓解策略**：
- **MVP 阶段用伪双 Agent 起步**：CTA 与 LCA 在同一进程中，通过函数调用而非消息队列通信
- **接口标准化**：用 Pydantic 定义 CTA → LCA 的消息契约（避免运行时类型错误）
- **超时保护**：LCA 调用超时 > 30s 自动降级为单 Agent 模式
- **单元测试覆盖**：CTA/LCA 接口单元测试 ≥ 90% 覆盖

**应急预案**：
- 若 MVP 阶段双 Agent 不可行 → 退化为单 Agent + 人工审核层
- 若互校循环延迟过高 → 引入异步互校（不阻塞实时干预）

**监控指标**：
- 双 Agent 接口错误率 ≤ 0.1%
- 互校循环延迟 P95 ≤ 5 秒
- LLM API 调用次数/学生/天 ≤ 50

### A2. CTA 5D 状态预测精度不足 🔴（对应 H1）

**触发条件**：
- CTA 5D 状态预测的 AUC < 0.75（[03-roadmap.md §2.3 H1 阈值](03-roadmap.md)）
- 5D 维度间相关性过高（Σ_θ 接近奇异矩阵）
- MIRT 参数冷启动失败（新学科/新学生）

**影响评估**：
- H1 验证失败 → M3 必须回溯到 M2 重新设计 CTA
- 学科诊断失去科学性（"伪置信"未被识别）
- 干预推荐基于错误状态 → 学习效果受损

**缓解策略**：
- **多基线对比**：CTA 5D vs IRT 1D vs BKT 单知识点 vs 人工标注（baseline 比较）
- **增量训练**：从 50 学生开始，逐步扩展到 100 → 500 → 1000（避免一次性大规模实验失败）
- **维度可降级**：若 5D AUC 不达标，可降级为 3D（K/P + Bloom）
- **人工校准集**：每月 10% 学生的 CTA 信念由学科专家审核

**应急预案**：
- 若 AUC < 0.65 → CTA 退化为 LLM-only + 教师审核（牺牲科学性换取可用性）
- 若 5D 失败 → 简化为 3D + Bloom（保留 ECOS 核心差异化）

**监控指标**：
- AUC（5D 状态预测下次表现）
- 维度相关性矩阵秩
- 人工审核分歧率 ≤ 15%

---

### A3. LCA 策略可解释性不足 🟡（对应 H3 部分）

**触发条件**：
- LCA 推荐的干预教师无法理解
- 学生/家长质疑"为什么给我这道题"
- 教师在班级管理时无法与 ECOS 推荐一致

**影响评估**：
- 教师/家长信任度下降（产品风险 B2）
- 学生主观体验差（"刷题感"）
- 商业模式 B2B 推广受阻（H5）

**缓解策略**：
- **rationale 输出**：每个干预附带自然语言理由（"你在二次函数的'分情况讨论'较弱，先做 3 道"）
- **教师后台**：教师可查看每个学生的干预历史 + 状态变化曲线
- **可视化**：5D 状态 + BloomProfile + TC 跨越用雷达图/进度条展示
- **可解释性 vs 性能权衡**：MVP 阶段宁可损失 10% 性能也要可解释

**应急预案**：
- 若教师普遍反馈"看不懂" → 增加教师培训 + 简化展示
- 若 rationale 质量差（LLM 解释不清） → 用模板化解释（预定义文本）

**监控指标**：
- 教师/家长满意度 ≥ 70%（问卷）
- rationale 可读性评分 ≥ 4/5
- 干预接受率 ≥ 60%

### A4. 双 Agent 互校无法抗幻觉 🔴（对应 H3）

**触发条件**：
- 双 Agent 信念校准度（ECE）> 0.10（[03-roadmap.md H3 阈值](03-roadmap.md)）
- LLM 解释文本中 misconception 命中率 < 0.5
- 互校循环陷入"CTA 错 → LCA 错 → CTA 跟着错"

**影响评估**：
- H3 验证失败 → M3 回溯
- 系统对外解释可信度崩塌
- 与竞品（Khanmigo/Squirrel AI）的科学化优势不复存在

**缓解策略**：
- **硬底线**：数学层 L0-L2 绝不用 LLM（v0.3.0 + v0.4.0 已确立）
- **LLM Critic 边界**：仅感知层 + 解释层 + Misconception 检测
- **人工审核触发**：当 CTA 信念置信度 < 0.6，自动转人工
- **互校对抗幻觉的 3 机制**（v2.0 §3.5）：CTA 保守 vs LCA 主动 + 数学严格 vs 教学法灵活 + 因果归因强制

**应急预案**：
- 若 ECE > 0.15 → 降低 LLM 介入深度（仅在感知层使用）
- 若互校失效 → 引入教师每周签字（牺牲自动化换可信度）

**监控指标**：
- ECE（双 Agent vs 人工标注）
- Misconception 检测 F1 ≥ 0.7
- 互校循环分歧率（CTA 与 LCA 判断不一致的比例）

### A5. Partial Credit 缺失 🔴（v0.52.2 发现，Phase 5 必修）

> **触发**：Bisen 2026-07-22 lbc001 PB-Q18 截图分析
> **依据**：[discussions/2026-07-22-partial-credit重大学术弊端发现.md](../../discussions/2026-07-22-partial-credit重大学术弊端发现.md)（8.2 KB）

**触发条件**：
- 学生答对 70% 但缺关键部分（如 I/O）
- 当前 MIRT 二元对错（`correct: true/false`）
- 70% 答对被按 0% 处理

**影响评估**：
- lbc001 PB-Q18 实际：学生答核心算法对，缺 `input()/print()` → AI 评判 `❌ correct: false`
- 5D 影响：K 维度 1.18 → 0.9638（跌 0.22），实际多跌 0.27
- L6 影响：多跌 0.2
- 整体：partial credit 缺失使 ECOS 在真实答题场景失真
- **不修 partial credit → ECOS 永远无法在真实答题场景应用**

**根因分析**：
- MIRT MAP 估计基于二元对错（response = 0/1）
- 不是 bug 而是设计选择——partial credit 改进需要重写 response scoring 函数
- 影响所有 5D 维度的真实评估

**缓解策略**：
- **v0.52.2 已存 AI reasoning**（Observation.ai_reasoning 字段）—— 留 Phase 5 训练用历史数据
- **Phase 5 v0.53.0 设计 partial credit 模型**：
  - Q 矩阵加权（5D 权重按 Bloom 层 + topic 重新分配）
  - AI 评判返回 `partial_score: float ∈ [0, 1]` 而非 `correct: bool`
  - MIRT 接受部分对（response scoring 改造）
- **短期**：v0.52.2 已存 ai_reasoning + judgement rationale 留历史数据

**应急预案**：
- 若 Phase 5 v0.53.0 partial credit 未实施 → 调整 MIRT 公式：partial_score 默认为 0.5
- 若 Q 矩阵结构不兼容 → 重做 Q 矩阵设计

**监控指标**：
- partial_score 分布（应近似均匀分布 0-1，而不是 0/1 二元）
- AI 评判 rationale 中 "部分对" 频率
- 5D 维度跌幅 vs 真实掌握度

**风险等级**：🔴 P0（Phase 5 必修，不修无法应用）

### A6. C/X 0 主导题 🟡（v0.52.1 发现）

> **触发**：Bisen 2026-07-22 lbc001 27 题测试
> **依据**：[discussions/2026-07-22-Phase5-Q矩阵CX重新设计路线图.md](../../discussions/2026-07-22-Phase5-Q矩阵CX重新设计路线图.md)（12.2 KB）

**触发条件**：
- 当前 lbc001 27-29 题均为 K/P/S 主导题（写代码题）
- C（Common mistakes / 调试题 / 错误分析）和 X（跨语言迁移）维度从未触发
- lbc001 实际数据：C=X=0.216 θ，SE=0.983，confidence=0.504

**影响评估**：
- 5D 评估实际 3D（C/X 永远显示低 confidence）
- 整体 5D 评估失真
- ECOS 核心价值"完整 5D 评估"受损
- H1 验证无法进行（5D 不完整）

**根因分析**：
- Q 矩阵设计 C/X 维度缺口（[research/90-mvp/python-basics-q-matrix-design.md §9.2](../90-mvp/python-basics-q-matrix-design.md) v0.52.1 扩充）
- 当前 Python 基础题库全是写代码题，缺调试/错误分析/跨语言类比

**缓解策略**：
- **方案 C（已落地 v0.52.1）**：C/X 标"待启用"灰底，不引入伪信号
- **Phase 5 v0.53.0 C 主导题 20+ 题**：
  - 调试题（debug 已知 bug）
  - 错误分析（code review / error classification）
  - code reading（理解代码而非编写）
  - debug strategy（系统化调试方法）
- **Phase 5 v0.54.0 X 主导题 20+ 题**：
  - Python↔JS / Java / C++ / Ruby 跨语言类比
  - 设计模式识别 / 算法跨语言实现
- **Phase 5 v0.55.0 X 维度 misconception 库**（M9-M16，8 条候选）

**方案选择（已决）**：
- ✅ 方案 C（标"待启用"灰底）—— v0.52.1 已落地
- ❌ 方案 A（0.10→0.20 伪信号污染）—— 不采用
- ❌ 方案 B（扩 40 题，一次性解决）—— 工作量过大

**应急预案**：
- 若 C/X 主导题设计难 → 简化 C 维度（仅调试题，10+ 题）
- 若 X 跨语言类比失败 → 聚焦 Python↔JS（最常用对比）

**监控指标**：
- lbc001 答 C/X 主导题次数
- C/X 维度 θ 变化（应逐步脱离 0.216）
- C/X confidence 是否突破 0.6

**风险等级**：🟡 P0（Phase 5 必修，但已用方案 C 兜底）

## B. 产品风险

### B1. Bloom 6 层在 K12 不适用 🟡（对应 H2）

**触发条件**：
- Bloom 6 层在初中数学的方差解释 < 60%（[03-roadmap.md H2 阈值](03-roadmap.md)）
- Evaluate/Create 层（L5-L6）数据稀疏（学生很少达到）
- 跨学科泛化能力差（数学可用，语文失效）

**影响评估**：
- H2 验证失败 → M3 必须回溯
- BloomProfile 失去预测力
- ECOS 与 Squirrel AI 等的差异化减弱

**缓解策略**：
- **MVP 阶段用 L1-L4**：暂不实现 Evaluate/Create（K12 实际很少达到）
- **学科适配**：数学 L1-L4，物理 L1-L4，语文可能只需 L1-L3
- **教师验证**：与学科教师确认 Bloom 分层是否合理

**应急预案**：
- 若 6 层方差解释 < 40% → 简化为 4 层（L1-L4）
- 若 Evaluate/Create 不可达 → 仅实现 L1-L4（MVP）

**监控指标**：
- BloomProfile 方差解释比例 ≥ 60%
- 各层级学生分布（避免 L5-L6 学生 < 5%）

### B2. 早期体验差（学生主观）🟡

**触发条件**：
- 学生首次使用后放弃率 > 30%
- 学生反馈"刷题感" / "比作业帮还烦"
- NPS（净推荐值）< 30

**影响评估**：
- 用户留存率下降（影响 H5）
- 口碑传播失效
- B2C 订阅模式不可持续

**缓解策略**：
- **认知学徒制 6 阶段**：避免纯刷题感，包含 Articulation（自我解释）+ Reflection（反思）
- **游戏化元素**：进度条、等级、勋章（不喧宾夺主）
- **首次体验优化**：第一次做题给出非常友好的反馈 + 引导
- **家长端可见**：学生知道"爸妈能看到我进步"

**应急预案**：
- 若放弃率 > 50% → 立即简化 UI + 增加游戏化
- 若 NPS < 20 → 重新设计引导流程

**监控指标**：
- 7 日留存率 ≥ 60%
- 30 日留存率 ≥ 40%
- NPS ≥ 30

### B3. 长期数据稀疏 🟡

**触发条件**：
- 单个学生累积数据 < 50 次作答（影响 BKT 校准）
- 跨学期数据缺失（暑假/寒假断档）
- 不同学科数据不平衡（数学多，物理少）

**影响评估**：
- CTA 状态估计精度下降
- 个性化推荐质量差
- 数据资产护城河形成慢（E4）

**缓解策略**：
- **冷启动**：用 K/P/S/C/X 默认先验 + 短期预测试
- **跨学期衰减模型**：长时间未用 ECOS 时按 BKT 衰减更新 P(L)
- **迁移学习**：从其他学生的同类数据初始化
- **不承诺早期个性化**：明确告诉家长/学生"前 4 周效果有限"

**应急预案**：
- 若数据稀疏 → 降低 BloomProfile 粒度（从 6 层到 4 层）
- 若跨学期断档 → 设计"复学测试"快速重新校准

**监控指标**：
- 平均每位学生作答次数/周 ≥ 10
- 跨学期留存率 ≥ 50%
- 数据稀疏学生比例 ≤ 20%

### B4. 学生数据采集质量差 🟡

**触发条件**：
- 学生应付式作答（乱选答案）
- 反思日志为空（学生不写）
- 答题时间不真实（秒选）

**影响评估**：
- CTA 状态估计被污染
- LLM Critic misconception 检测失效
- 数据资产价值降低

**缓解策略**：
- **题型多样化**：不能只看对错，看过程（解答步骤）
- **强制反思节点**：每 N 道题必须做一次反思（不是每次）
- **异常检测**：秒选 + 全部选 C 自动标记为低质量
- **质量权重**：低质量数据在 BKT 更新中降权

**应急预案**：
- 若质量问题严重 → 增加人工审核样本
- 若反思日志普遍为空 → 改为选择题形式（结构化反思）

**监控指标**：
- 反思日志完成率 ≥ 50%
- 异常作答比例 ≤ 10%
- 数据质量评分（人工审核样本）≥ 4/5

### B5. 无 pytest 自动化测试 🟡（lbc001 单用户手动测试）

> **触发**：v0.53.1 审查报告 §10 测试基础设施评估
> **依据**：[07-project-comprehensive-audit-2026-07-22.md §10](07-project-comprehensive-audit-2026-07-22.md)

**触发条件**：
- 当前无 pytest 单元测试套件
- 测试依赖 lbc001 单用户手动测试（27-29 题）
- 任何字段恢复改动都可能引入新 BUG

**影响评估**：
- 回归靠手动，效率低
- DB 恢复历史教训 4 次漏（import json / tc_states / trajectory / item_params）
- 字段 schema 变更无测试保护
- Phase 5 partial credit 实施无回归保护

**根因分析**：
- Phase 4 资源聚焦 CTA + UI
- 自动化测试编写成本高
- 单用户 lbc001 测试能发现关键 BUG

**缓解策略**：
- **短期**：保持 lbc001 单用户手动测试
- **中期**：Phase 5 v0.55.0 加 pytest 单元测试套件：
  - CTA state update（MIRT MAP / Bloom 累积 / TC 状态机）
  - DB 恢复（5 维状态 + θ_cov + Bloom + TC + trajectory + response_history + misconception_history + item_params）
  - LLM 调用 mock（避免 LLM 依赖）
- **长期**：加 GitHub Actions CI + pre-commit hook

**应急预案**：
- 若回归 BUG 频发 → 暂停功能开发，加测试套件
- 若测试覆盖不足 → 优先核心模块（CTA + persistence）

**监控指标**：
- pytest 单元测试覆盖率（目标 ≥ 80%）
- 回归 BUG 频次（应逐步降低）
- 手动测试 vs 自动化测试比例

**风险等级**：🟡 中（短期可接受，Phase 5+ 必加）

## C. 教育专业风险

**监控指标**：

- 防御性自检 5 项跑过频率
- 沉默失败 `except: pass` 数量

## D. 伦理与法律风险

### D1. 未成年人数据合规 🔴

**触发条件**：
- 违反《个人信息保护法》（中国 2021 实施）
- 违反《儿童个人信息网络保护规定》
- 家长投诉 / 监管处罚

**影响评估**：
- 法律风险（罚款 + 业务暂停）
- 声誉风险（教育行业敏感）
- 商业模式无法启动

**缓解策略**：
- **最小化数据采集**：只采集必要的学习数据（不做行为画像）
- **家长明确同意**：注册流程必须家长双重确认
- **数据本地化**：学生数据存中国境内服务器
- **端侧计算**：敏感数据（姓名、学校）端侧处理
- **差分隐私**：聚合数据发布时加入噪声
- **第三方审计**：每年邀请第三方做数据合规审计

**应急预案**：
- 若被监管约谈 → 立即暂停 + 主动汇报 + 整改
- 若发生数据泄露 → 24 小时内通报 + 启动应急响应

**监控指标**：
- 数据合规审计通过率 = 100%
- 家长撤回率 ≤ 5%
- 数据泄露事件数 = 0

- H7 验证失败 → 数据资产护城河不成立
- ECOS 失去长期竞争优势
- 投资价值降低

**缓解策略**：
- **早期数据收集**：从 MVP 开始就重视数据质量（B4 风险缓解）
- **跨学段衔接**：与高中合作，承接 ECOS 初中毕业生
- **数据资产化**：建立数据资产估值模型，定期对外披露
- **学术研究**：用数据发表论文，建立"ECOS 数据"品牌

**应急预案**：
- 若 3 年数据 < 3000 → 聚焦单学段（初中 3 年）+ 接受数据规模小
- 若流失率 > 70% → 重新设计留存机制

**监控指标**：
- 3 年纵向数据规模 ≥ 5000 学生
- 学生年均流失率 ≤ 30%
- 跨学段衔接率 ≥ 40%

## F. 风险监控与应对机制

### F1. 风险监控指标看板

| **法律** | 合规审计 / 数据泄露事件 | 每季度 |
| **商业** | 转化率 / 客单价 / 决策周期 | 每月 |

**红线指标**（一旦触发必须立即响应）：
- 数据泄露事件 ≥ 1
- 合规审计不通过
- 7 日留存率 < 30%
- LLM API 成本超预算 200%
- 监管约谈 / 媒体负面报道

### F2. 风险升级流程

```
Level 1（个人处理）
    ↓ 24 小时内未解决
Level 2（团队处理）
    ↓ 72 小时内未解决
Level 3（创始人决策）
    ↓ 影响 Phase 进度
Level 4（暂停 + 回溯）
```

### F3. 应急预案触发条件

| 触发条件 | 应急预案 |
|---|---|
| H1（H3 后）失败：AUC < 0.65 | CTA 退化为 LLM-only + 教师审核 |
| H2（H3 后）失败：Bloom 方差解释 < 40% | 简化为 L1-L4 |
| H3（H3 后）失败：ECE > 0.15 | 降低 LLM 介入深度 |
| 数据泄露 | 24 小时通报 + 应急响应 |
| 监管约谈 | 暂停 + 主动汇报 + 整改 |
| 商业模式连续 2 季度失败 | 商业模式大调整 |
