## 起点：上轮探讨遗留的两个方向

2026-07-10 已完成分析对比，结论是**推荐方向 B（混合架构）+ 5-8 题 warm-up 窗口**：

| 维度 | 方向 A：纯两阶段 | 方向 B：混合架构 |
|---|---|---|
| 测量学叙事 | 干净（CAT-style SE<0.3）| 弱（连续估计）|
| 学习体验 | 8-15 题冷启动劝退 | 零冷启动 |
| Misconception 检测 | 漏掉过程涌现的 | 完整捕获 |
| Q 矩阵需求 | 切两池，每池统计效力不足 | 一池，每题 4 重价值 |
| 架构契合度 | 低（需引入 phase 状态机）| 高（80% 已是混合形态）|

**本轮关键问题不再是 A vs B，而是 B 落地时还要解决什么。**

---

## 方向 A vs B 的最终决策：**选 B**

**结论**：ECOS Phase 4（Product Demo 完整化）采用方向 B（混合架构）+ 5 题 warm-up 窗口（不是 8-15 题）。

### 4 条核心理由

1. **K12 自学场景的核心 KPI 是"学生愿不愿意继续打开"**——8-15 题纯测试是最高流失率姿势开局
2. **Misconception 是过程涌现的**——两阶段只能测"进入时已有"；过程中跨 TC 临界点形成的新错误图式会被漏掉
3. **24 题 Q-matrix 切两池都不够**——B 方案每题 4 重价值（诊断 / 学习 / TC 推进 / misconception 检测）杠杆率最高
4. **现有 BeliefEngine 已是 80% 混合形态**——缺的不是诊断能力，是**自适应选题层**把诊断信号反哺到选题；这是低成本补完，不是架构重写

测量叙事的损失是有的，但**那是能用"探针题 + 置信度曲线"以后补回的**；冷启动劝退是**当下直接砍留存**的。

## 5 个开放问题的最终决策

### 问题 1：ECOS 定位是 B 端测量还是 C 端学习？

**决策：C 端学习产品为先，B 端机构作为远期延伸。**

| 维度 | 决策 |
|---|---|
| 核心迭代围绕 | C 端学生体验 |
| 数据层 | 留"B 端机构"接口（学生 ID 支持一对多）|
| 报告层 | dashboard 留"导出学习报告"按钮 |
| UI 层 | **不做任何 C 端教师/家长 UI** |

**判断标准**：任何 UI 决策先问"这是给学生自己看的,还是给老师看的?"——只要是后者,推到 Phase 5+。

### 问题 2：是否接受 warm-up 窗口？

**决策：接受，但要"无感"**。

| 维度 | 决策 |
|---|---|
| 题目数 | **5 题**（不是 8-15）|
| UI 文案 | "正在熟悉你的节奏"——不显示具体数字 |
| 过渡方式 | 5 题后**平滑过渡**，不要"诊断结束"的硬切换 |
| 学生感知 | 学生不该知道刚才那 5 题是 warm-up |

### 问题 3：探针题的频率和强度？

**决策：每 8-10 题穿插 1 道,无痕,不计学习时长。**

| 维度 | 决策 |
|---|---|
| 频率 | 8-10 题（太密干扰学习,太疏失去校准价值）|
| 强度 | **学生无感**——探针题表面看是正常学习题,但选题时优先选"能压低某维度 SE"的那道 |
| 标注 | **不显式标注** |
| 学习时长 | **不计**（避免家长觉得孩子在"被测"）|

### 问题 4：目标态来源？

**决策：Phase 4 用 Bloom 6 级自动梯度,Phase 5 再做个性化。**

| 维度 | 决策 |
|---|---|
| Phase 4 范围 | 每节课按"当前 Bloom 层 → 上一层"作为目标 |
| 不做的事 | 不要让用户/家长设定目标（**降低使用门槛**）|
| Phase 5+ | 加"教师/家长可设定个性化目标"作为高级功能 |

### 问题 5：是否补完 `target_theta` / `target_bloom_profile`？

**决策：Phase 4 不补,Phase 5 必须补。**

| 维度 | 决策 |
|---|---|
| Phase 4 范围 | dashboard 加"当前 Bloom 层 + 距下一层 Δ"——1 天工作量 |
| 显式数据结构 | 推迟到 Phase 5（`target_theta` / `target_bloom_profile` 数据结构）|
| 理由 | Phase 4 核心是"产品 Demo 完整化",引入会拖慢节奏 |

## 产品定位的最关键决策：先 A 后 C

讨论最后,Bisen 回答了"ECOS 最想被谁用"的反问：

> "从我本心出发,希望是 A(学生自己打开学,完全自主),但从项目商业化角度,好像又不能回避 C(学校采购当课后练习)的可能性。"

### 决策:**先 A,做兼容 C 的设计,但不主动做 C 端功能**

**一句话原则**：**A 跑通,做 C 是降维;先做 C,可能再也回不去 A。**

### 3 条核心理由

1. **ECOS 的核心研究命题要求 A**——ECOS 探索"AI 能否持续理解并帮助 K12 学生成长",自主学习场景最能验证这个假设。被学校采购后,产品方向就从"AI 理解学生"变成"AI 服务学校",核心命题就变了。

2. **做好 A 转 C 是降维,反过来不行**
   - A → C:把"自主学习"包装成"机构辅助产品",数据、报告、班级管理都是已有能力的封装
   - C → A:学校采购会倒逼你做"班级管理、教研对接、教材合规",学生自主性会被结构性削弱

3. **A 跑通后,自然会有 C 来找你**——路径是：**真实学生数据 → 案例研究 → 方法论沉淀 → B 端机构主动询单**。这条路比"先做 C 端功能等采购方来"健康得多——卖的是"方法论 + 案例",不是"软件 license",议价权完全不同。

### 兼容性接口（数据层留,UI 不做）

| 维度 | 现在做 | 不做 |
|---|---|---|
| 数据 | 学生 ID 支持"一对多"（为"班级"留位）| 班级管理 UI |
| 报告 | dashboard 留"导出学习报告"按钮 | 教研报告模板 |
| 进度曲线 | 形态适合"对外汇报" | 教师视角 dashboard |
| 测量叙事 | 探针题 + 置信度曲线 | 标准化测评报告 |

**关键判断**：这些接口的总成本 < 5%,但保留了"将来转 C"的可能性。

### 反向决策锚点（什么时候可以开始认真做 C）

出现以下任一情况,就可以开始认真做 C：

- A 端获客成本 > 学生 LTV（说明 C 端市场还没准备好）
- B 端机构**主动找上门**且需求合理（说明市场验证了）
- 12 个月内 A 端跑不到 100 个真实学生（说明 C 端是更现实路径）

否则,坚持 A。
