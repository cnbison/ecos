## 1. 关键发现: C 主导题让 C 维度"真实化"

### 1.1 表面现象

| 指标 | 改造前 | 改造后 | 变化 |
|------|--------|--------|------|

### 1.2 根因分析: MIRT MAP 估计的"重新校准"

**改造前 (0.2145)**:
- lbc001 33 题全是 K/P/S 主导 (PB-Q01~PB-Q26)
- C 维度 θ=0.2145 是**从 K/P/S 协方差推断的"占位值"**
- C 维度**从未被真实数据校准**——ECOS 假装知道 lbc001 C 能力, 实际不知道

**改造后 (-0.07)**:
- lbc001 5 道 C 主导题 (PB-C01~PB-C05), 模拟 partial credit 评分:
  - PB-C01 (range 越界): 0.6 (识别+修复, 没解释)
  - PB-C02 (continue 诱饵): 0.6 (识别陷阱, 漏关键点)
  - PB-C03 (TypeError): 0.6 (识别+单修法)
  - PB-C04 (scope): 1.0 (完整答对)
  - PB-C05 (fib 调试): 0.6 (多方法没排序)
- **平均 partial credit 0.68** (68%)
- MIRT MAP 估计判断: lbc001 之前 C=0.21 应该是高能力, 但实际 5 题只 0.68 表现 → **调低 C 维度到 -0.07**

**这是 MIRT MAP 估计的"诚实"行为**:
- 之前 C 维度 0.21 是**虚高**——ECOS 没 C 维度数据, 假装 C 维度跟 K/P/S 类似
- 现在 C 主导题让 C 维度**真实化**——lbc001 实际 C 能力中等偏下 (-0.07)
- C 维度从"占位 0.21" → "真实 -0.07", **不是跌, 是"诚实化"**

## 4. C 主导题 L1/L2 缺口分析 (Bisen 2026-07-23 提问)

### 4.1 实际数据: 20 道 C 主导题 Bloom 层分布

| Bloom 层 | 数量 | 占比 |
|----------|------|------|
| L1 (Remember) | **0** | 0% |
| L2 (Understand) | **0** | 0% |
| L3 (Apply) | 5 | 25% |
| L4 (Analyze) | 6 | 30% |
| L5 (Evaluate) | 4 | 20% |
| L6 (Create) | 5 | 25% |
| **合计** | **20** | 100% |

**C 主导题只涵盖 L3-L6 (4 层), 缺 L1/L2 (2 层)** — Bisen 提问准确.

### 4.2 为什么缺 L1/L2? 5 个原因

#### 原因 1: 4 类题型天然适合 L3-L6

| 4 类 C 主导题型 | 典型 Bloom 层 |
|----------------|----------------|
| 调试题 (Debug Bug) | L4 (Analyze) 找位置 + L5 (Evaluate) 评估原因 |
| 错误分析 (Error Analysis) | L4-L5 推理错误原因 |
| 代码阅读 (Code Reading) | L2-L3 理解/应用 |
| 调试策略 (Debug Strategy) | L5-L6 规划/创造 |

**L1 (Remember)** "记忆" + **L2 (Understand)** "理解" 偏向 K 主导 (知识理解) 而非 C 主导 (错误识别).

#### 原因 2: v0.54.0-b 设计 5 道时已隐含这一选择

PB-C01~PB-C05 (v0.54.0-b) 全部 L3-L5, 没 L1/L2:
- PB-C01 (L3 loops 调试题)
- PB-C02 (L4 loops 调试题 诱饵)
- PB-C03 (L5 functions 错误分析)
- PB-C04 (L5 scope 代码阅读)
- PB-C05 (L4 recursion 调试策略)

**v0.54.1 扩 15 道时按 5 topic × 4 Bloom 层 (L3-L6) 均匀分布, 延续了这一选择**.

#### 原因 3: L1/L2 偏向 K 主导, 跟 C 主导题类型重叠

| Bloom 层 | K 主导题典型 | C 主导题典型 |
|----------|--------------|--------------|
| L1 (Remember) | 变量名/语法/标准库 | "这段代码错在哪 (记忆类错误)" — 偏 K |
| L2 (Understand) | 理解代码输出/行为 | "识别错误类型" — 偏 C |
| L3 (Apply) | 写代码/套公式 | 调试题/错误分析 — 偏 C |
| L4-L6 | 高级应用/分析/评价/创造 | 同上 — 偏 C |

**L1/L2 C 主导题跟 K 主导题边界模糊**, 强行设计可能让 C 维度评估跟 K 维度重复.

#### 原因 4: lbc001 实际能力 L1/L2 主要由 K 主导题覆盖

lbc001 33 题 PB-Q 分布:
- L1 (Remember): 4 题 (PB-Q01, 05, 09, 12)
- L2 (Understand): 5 题 (PB-Q02, 06, 10, 13, 15)
- L3 (Apply): 5 题 (PB-Q03, 07, 11, 14, 16)
- L4 (Analyze): 2 题 (PB-Q04, 08)
- L5 (Evaluate): 5 题 (PB-Q17, 19, 21, 23, 25)
- L6 (Create): 6 题 (PB-Q18, 20, 22, 24, 26)

L1/L2 共 9 题 (27% 占比), 全是 K 主导 (输出什么/写出代码). **L1/L2 调试类题型缺失**.

#### 原因 5: v0.54.1 时间紧, L1/L2 留 v0.54.2+ 补充

v0.54.1-b 时间窗 (Bisen 21:13 决定 → 21:21 完成 20 道), 优先 L3-L6 4 类典型题型.
L1/L2 调试题 (如 "以下代码错在哪 - 记忆类错误") 设计需要更多讨论, 留 v0.54.2+.

### 4.3 L1/L2 C 主导题补设计建议 (v0.54.2+ 可选)

| 题目 | Bloom | topic | 题型 | 说明 |
|------|-------|-------|------|------|
| PB-C21 (L1 loops 记忆) | L1 | loops | 调试题(记忆) | "range(1, 5) 包不包括 5" — 记忆 + 简单调试 |
| PB-C22 (L2 variables 理解) | L2 | variables | 错误分析(理解) | "以下代码错在哪 (类型/语法错误)" — 理解类错误 |
| PB-C23 (L2 functions 理解) | L2 | functions | 代码阅读(理解) | "以下函数返回什么" — 输出识别错误 |
| PB-C24 (L1 scope 记忆) | L1 | scope | 调试题(记忆) | "global x 是声明全局变量吗" — 记忆 |

**4 道 L1/L2 C 主导题补充 v0.54.1 缺口**, 让 C 主导题覆盖 6 层 Bloom (L1-L6).

### 4.4 缺口的实际影响评估

| 维度 | 评估 |
|------|------|
| **C 维度评估完整性** | 影响低 — L1/L2 C 主导题跟 K 主导题重叠, 补充价值有限 |
| **5D 联合 MIRT 估计** | 影响低 — L1/L2 题型简单, 不会让 MIRT 协方差大波动 |
| **lbc001 C 能力评估** | 影响低 — lbc001 在 L3-L6 C 主导题上 -0.07 评估已稳定 |
| **学术严谨性** | 影响中 — C 主导题没覆盖 L1/L2 是设计缺口, 写论文时要说明 |

**结论**: L1/L2 缺口**实际影响小**, v0.54.2+ 补充 4 道 L1/L2 C 主导题即可, 不阻塞 Phase 5 推进.
