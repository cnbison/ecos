# 探讨记录：诊断与教学的相位分离

**日期**：2026-07-10
**参与者**：Bisen、Claude
**主题**：BeliefEngine 架构——是否应分离"初始认知状态诊断"与"过程状态演变"

---

## 探讨背景

本次探讨从一次 UI 字段解释切入。Bisen 打开 Product Demo 学生端进入学习后，注意到顶部三联指标：

- `Bloom: REMEMBER`（当前主导认知层）
- `C折扣: 1.00`（元认知维度 misconception 折扣）
- `总置信度: 0.000`（CTA 整体置信度）

字段含义解释完后，Bisen 顺着 `总置信度: 0.000` 这一现象追问：

> "我理解你的业务逻辑是把判别学习者的初始认知状态和过程改变都放到一起了是吗？而不是先用一些题目来测试评判学习者当下的认知状态。然后根据目标状态来针对性干预从而实现状态的演变？"

这个问题触及 Phase 4 Product Demo 的核心架构选择，因此展开本轮深度探讨。

---

## Bisen 的核心质疑

当前 BeliefEngine 把**诊断**（估计学习者当前认知状态）和**教学**（追踪状态变化并干预）合并到单一在线更新流。每答一题，`BeliefEngine.update(state, obs)` 一口气完成：BKT 更新、MIRT MAP 估计、Bloom profile 更新、LLM Critic misconception 检测、TC 状态检测、整体置信度计算。

Bisen 指出这与教育测量学的经典做法（先做 calibration / placement test，再做 targeted instruction）不一致。隐含的问题是：

- 是否应该先用一个校准过的题库跑一次预测试，**专门**估计初始状态？
- 是否应该有显式的目标态（target theta / target bloom layer），并据此驱动题目选择与干预？
- 当前实现是否在结构上漏掉了什么？

---

## 代码层分析：当前实际发生什么

### 1. 单一在线更新流，没有相位分离

`BeliefEngine.update()`（`ecos/cta/belief_engine.py:175-253`）每次答题被调用一次，内部 10 个步骤连续执行。新生初始态 `theta=0, confidence=0`（`belief_state.py:150`）。

前 30 题的回复**同时承担两个角色**：
- 估计初始能力（诊断）
- 记录学习过程（教学）

MIRT 置信度 `min(1.0, len(history)/30.0)`（line 204）意味着前 30 题 SE 都很高，估计噪声大。但这些题目又同时被当成学习事件写进 trajectory，无法事后区分"这 30 题里有多少是诊断、有多少是教学"。

### 2. 题目选择是随机的，不是自适应的

`select_question_for_student()`（`web/api/qmatrix.py:46-67`）的 docstring 写"优先选学生薄弱 topic + 低 Bloom 层题目"，但实际代码：

```python
unanswered = [p for p in all_probs if p["problem_id"] not in answered_ids]
return random.choice(unanswered)
```

**就是随机抽取**。没有根据 `state.theta_K/P/S/C/X` 或 `bloom_profile.dominant` 做选题决策。BeliefEngine 输出的诊断信号没有反哺到题目选择层。

### 3. 没有显式目标态

grep `ecos/cta/` 下的 `target/goal/aim/objective/delta_state`，唯一一处 `bloom_target` 出现在 `llm_critic/explanation.py:163`，且仅作为生成解释文本的提示词（"如果没到 L4，目标设为下一层"）。

没有 `target_bloom_profile` 或 `target_theta_5d` 数据结构，也没有"当前态到目标态距离"的计算。

CLAUDE.md 中提到的 `Bloom Goal Space`（研究纲领核心概念）在当前代码里**没有对应实现**——它是研究阶段的概念，未落到 Phase 4 状态机里。

---

## 研究纲领层分析：Bloom Goal Space 隐含什么

研究纲领里的 `Bloom Goal Space` 隐含两阶段架构：

- **Phase A：诊断**——用校准过的题库（Q-matrix + MIRT a/d 参数已具备）跑一次短测，把 θ 的 SE 压低到可信阈值
- **Phase B：靶向干预**——根据 `state.target_theta - state.theta` 选题（LCA 派任务，干预 misconception，提升 Bloom 层）

当前 M2 实现的差距：

- ❌ 题目选择自适应
- ❌ 显式目标态结构
- ❌ 诊断 vs 教学相位分离
- ❌ 干预前后 theta 增量归因（trajectory 有快照但无"Δ 由哪道干预题贡献"的因果标注）

---

## 两种方向：利弊对比

### 方向 A：纯两阶段（Diagnostic → Targeted Instruction）

**利**
- 测量学叙事干净（CAT-style 可声明 SE<0.3）
- 阶段边界清晰，家长/教师易理解（"现在在测 / 现在在学"）
- 所有维度在诊断池都能被探测
- 干预 ROI 可量化（"教学阶段每题平均让 θ 提升 X"）

**弊**
- 8-15 题冷启动 → K12 自学产品的"测试感"劝退
- 24 题 Q-matrix 切两池，每池统计效力不足
- **结构性漏掉"学习中涌现的新 misconception"**（最严重的一条）
- 与现有 BeliefEngine 架构契合度低（需引入 phase 状态机，改 update 流）
- 用户感知上"测完才能学"的延迟感强

### 方向 B：混合架构（持续诊断 + 自适应教学）

**利**
- 第 1 题就是学习题，零冷启动
- 每题 4 重价值（诊断 / 学习 / misconception 检测 / TC 推进）
- 自适应选题可显式针对 SE 最大维度 + Bloom 层差 + 弱 topic
- 目标态可显式呈现（dashboard 显示"距 L3 还有 ΔX"）
- 与现有架构契合度高，改动量小

**弊**
- 前 5-8 题状态估计噪声大，dashboard 数字会跳变
- "目标达成"没有清晰时刻，是渐进的
- 测量学叙事弱——对学校采购方只能说"连续估计"
- 强维度永远不被主动探测 → 验证盲区

---

## 推荐与理由

**推荐方向 B（混合架构）+ 5-8 题 warm-up 窗口**。前 5-8 题专注把 SE 压下来（不显示具体数字，显示"正在评估你的起点"），之后切换到自适应教学 + 持续诊断。

### 4 条核心理由

**1. 用户场景决定首要矛盾**

K12 自学产品面对的是"打开 App 想学 Python 的学生"，不是"被老师安排做分级测试的学生"。冷启动 8-15 题诊断会制造测试感、浪费学习机会、劝退不愿被测试的学生。

**2. Misconception 是过程中涌现的**

现有 `tc_detector` 和 `llm_critic/misconception_detector` 的核心假设就是"误概念在学习中动态形成"。两阶段预测试只能抓到学生**进入时已有**的 misconception；教学阶段新涌现的（典型：跨过 TC 临界点时形成新错误图式）会被漏掉。这是结构性损失。

**3. 24 题 Q-matrix 撑不起纯两阶段**

切两池，每池统计效力都不足。混合架构下每题同时贡献诊断 + 学习 + TC 推进，杠杆率最高。

**4. 现有 BeliefEngine 已是 80% 混合形态**

update 流每次都做 MIRT MAP、Bloom 更新、misconception 检测、TC 推进——诊断工作一直在做。缺的不是诊断能力，是**自适应选题层**把诊断信号反哺到选题。这是低成本补完，不是架构重写。

### 必须配套的 3 项缓解措施

- **置信度 UI 透明化**：置信度 < 0.5 时 dashboard 数字变灰 + tooltip 解释
- **Milestone 庆祝**：跨 Bloom 层时弹提示
- **定期探针题**：每 5-10 题穿插 1 道强维度探针题，验证估计准确性

## 决定取舍的关键问题

**ECOS 最终按"测量产品"卖，还是按"学习产品"卖？**

- B 端学校/教育局采购 → 测量叙事权重高 → 方向 A 或带显式诊断窗口的混合
- C 端学生/家长付费 → 学习体验权重高 → 方向 B + 探针题

如果定位还在摇摆：方向 B + warm-up + 探针题 + 置信度 UI 透明化——既保学习体验，又给 B 端测量叙事留接口。
