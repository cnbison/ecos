**v0.53.0 双目标**:
1. **Partial Credit 必修** (Bisen 必解, 不能推迟)
2. **C 主导题扩 20+ 题** (原计划)

#### Partial Credit 实施

**LLM 评判输出 `score: 0.0-1.0`, 不用 `correct: bool`**

新评判 prompt:
```json
{
    "score": 0.7,        // 0.0-1.0, 部分对
    "correct": false,    // score >= 0.6 仍算对 (兼容)
    "reasoning": "核心算法对, 缺 I/O",
    "error_type": "incomplete",  // 粗心/不理解/概念混淆/API记错/不完整
    "missing_components": ["input()", "print()"],
    "demonstrated_skills": ["arithmetic_extraction", "reverse_composition"]
}
```

新 MIRT 更新 (partial credit):
- `theta` 更新按 score 加权 (score=0.7 时 K 涨 0.7×0.05 而非 0 或 0.22)
- `cov` 更新按 (1 - score) 加噪声
- `dim.confidence` 公式不变 (仍是 1/(1+SE))

新 Bloom 更新 (按 demonstrated_skills):
- 即使 correct=false, 如 demonstrated_skills 含 L6 元素, L6 confidence 不跌
- 如 missing_components 是 L1-L2 (I/O 完整性), 不影响 L6

新 TC 更新 (按 error_type):
- careless → TC 不变
- incomplete → TC 不变 (缺 I/O 不是 TC 跨越)
- misconception → TC progress 跌 + misconception 触发
- unfamiliar → TC progress 跌
- api_misremember → TC progress 微跌

**触发条件 (Partial Credit 必修)**: lbc001 答 30+ 题, 有足够 partial credit 训练数据 (response_history 现在已存 ai_reasoning, v0.52.2 起的)

### v0.53.0 - C 主导题扩 20+ 题

**目标**：让 C 维度真正可评估

**数量**：20 题
**topic 分布**：5 topic × 4 题
**Bloom 分布**：L1-L4 (基础元认知) × 4 题 + L5-L6 (调试) × 4 题

**C 主导题模板**：

| 题型 | 示例 | Bloom | C a_specialized |
|------|------|-------|-----------------|
| **调试题** (给 bug 代码, 找出 bug) | `x = 0; while x < 5: print(x) # 死循环, 缺 x += 1` | L4-L5 | 0.50-0.70 |
| **错误分析题** (给常见错误解法, 解释为啥错) | "学生答 range(5) 包括 5, 错在哪?" | L3 | 0.50 |
| **code reading** (给代码, 解释执行结果) | `def f(x): return x*2; print(f(f(3)))` | L2-L4 | 0.50 |
| **debug strategy** (设计 test case 找 bug) | "为这个函数设计 3 个 test 找 bug" | L5-L6 | 0.60-0.80 |

**a_specialized 设计原则**：
- C 维度：0.50-0.80 (主测)
- 其他维度：≤ 0.20 (弱信号)

**Misconception 关联**：现有 M1-M8 (Python misconception 库) 已经覆盖 C 维度，新题应至少包含 5-8 条 misconception 触发题。

**LCA 选题权重**：当 C theta SE > 0.7 时优先选 C 主导题 (类似 v0.42.0 W2 自适应权重)。

**触发条件 (何时启动 v0.53.0)**：
- lbc001 答满 30+ 题 (有足够 K/P/S 数据)
- Bisen 启动决策

| **模式抽取** (给 Python + 另一语言代码, 问共同模式) | "Python `for i in [1,2,3]: print(i)` 与 JS `for(let i of [1,2,3]) { console.log(i) }` 的共同点是?" | L3-L4 | 0.50 |

## 3. 配套工具与流程

### LLM 生成 prompt 模板

```python
# C 主导题生成 prompt
PROMPT_C = """
请生成 1 道 Python {topic} C 维度主导题, Bloom L{level}。
C 维度 (条件/元认知) 是主导维度, a_specialized 应该是 C 主导 (C ≥ 0.5, 其他 ≤ 0.2)。

题目类型: {调试题 | 错误分析题 | code reading | debug strategy}
Bloom 等级: L{level} (1=Remember, 2=Understand, 3=Apply, 4=Analyze, 5=Evaluate, 6=Create)
Topic: {topic} ({skill_name} 领域)
难度: {easy | medium | hard}

返回严格 JSON:
{{
  "problem_id": "PB-CR{xx}",
  "problem_text": "...",
  "correct_answer": "...",
  "bloom_layer_observed": "L{level}",
  "topic": "{topic}",
  "skill_name": "...",
  "a_specialized": [K, P, S, C, X],
  "mirt_params": {{"difficulty": ..., "discrimination": ..., "guessing": ...}},
  "misconceptions": ["M1", ...]  // 触发哪些 misconception (M1-M8 for C 维度)
}}
"""
```

### 审核工作流

```
LLM 生成 → JSON 校验 → Bisen 人工审题 → 入库
```

- **LLM 生成**：用 Claude / GPT-4, 一次生成 1 题, 跑 20-30 次
- **JSON 校验脚本** (`scripts/validate_qmatrix.py`)：
  - 校验 a_specialized 各维度都在 0-1 之间
  - 校验 a_specialized 主导维度 ≥ 0.5
  - 校验 a_specialized sum ≤ 2.0 (避免冲突)
  - 校验 problem_text / correct_answer 非空
- **Bisen 人工审题**：
  - 题目质量 (语法/逻辑/难度)
  - 正确答案是否唯一
  - a_specialized 是否合理 (主导维度确实主测该题)
  - misconception 标注是否准确
- **入库**：`data/python_basics_q_matrix.json` 直接 append

### CI gate (CLAUDE.md 防御性自检)

- 改 Q 矩阵前: `python scripts/validate_qmatrix.py` 校验
- 改 a_specialized 前: 跑 `git grep -nE 'a_specialized'` 看全工程引用
- commit 时: 在 commit message 列出每题 problem_id + a_specialized 决策理由
