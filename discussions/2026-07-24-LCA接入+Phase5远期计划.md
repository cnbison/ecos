## 1. 背景与根因

### 1.1 A9 风险解读

`research/00-overview/04-risks.md` §A9 标题"**LCA 未实施 🟠（仅有脚手架）**"——LCA 框架代码（`ecos/lca/`，2026-07-03 v0.30.0 完成）写好了，但**没接进主循环**：

- `web/api/belief.py` grep "LCA" 0 匹配
- 所有"下一步该做什么"实际是 CTA 状态估计 + 简单选题加权
- ECOS 核心价值"理解 + 改变"只实现了一半
- A10（双 Agent 互校）跟 A9 强相关——没 LCA 就没互校对象

### 1.2 v0.56.0 接入前的现状盘点

| 组件 | 状态 | 证据 |
|------|------|------|
| `LCAEngine` 主类 | ✅ 写好 | `ecos/lca/orchestrator.py` 完整 6-step 流程 |
| 5 类干预 + 4 级 CLT + 6 阶段 CA | ✅ | `intervention.py` |
| L3 自适应（CLT/Bjork/Scaffolding） | ✅ | `l3_selection/` |

| RationaleGenerator（LLM 表达层） | ✅ | `rationale/generator.py` |
| **LCAEngine 接入主循环** | ❌ **零** | belief.py 完全没调 LCA |
| **LinUCB Reward 信号** | ❌ | `engine.update()` 写了但没人在调 |
| **LCA 持久化** | ❌ | `intervention_history` 是 in-memory dict |

**结论**：v0.56.0 真正的工作**不是"造 LCA"，而是"接通 LCA + 实施互校 + 让 H3 跑得起来"**。

## 2. v0.56.0 实施回顾（commit `50fc332`）

### 2.1 Definition of Done vs 实际

- [x] `web/api/app.py` 接入 `LCAEngine` 单例 → ✅ `web/api/lca.py` 包装层

### 2.2 关键技术决策

2. **Reward 简化版**：只用 K 维度变化（score）+ bloom 层是否答对，不用 5D 全量 state_delta（v0.57.0+ 升级）
3. **In-memory 状态**：`_last_intervention` / `_update_count` 是模块级 dict，**进程重启即丢**——v0.57.0 持久化
4. **测试 38/38 通过**：16 新增 + 22 原有

## 3. v0.57.0 - v0.59.0 详细计划

> **核心思路**：LCA 框架代码 85% 已写好。剩下 15% 是"接电源 + 装记忆 + 配搭档 + 验证搭档有用"——**4 步走，2-3 周，不画饼**。

### 3.1 v0.57.0 — LCA 持久化（让 LinUCB 跨重启学习）

**目标**：解决 `intervention_history` in-memory 问题，LinUCB 参数能跨进程恢复。

**触发条件**：v0.56.0 跑通 1 周 lbc001 数据 + 确认 LCA 决策合理后启动。

**Definition of Done**：
- [ ] 新增 `ecos/persistence/lca_store.py`：序列化 `intervention_history` + `LinUCB.A/b 矩阵` 到 SQLite（参照 `belief.py` 的持久化模式）
- [ ] `_get_or_create_lca_state(student_id)` 恢复流程（**CLAUDE.md 防御性自检 [5] 必跑**——一次性把 6 个关键字段全列出来：intervention_history / bandit.A / bandit.b / bandit.arm_pull_counts / last_intervention / update_count，别像 4 次漏字段那样分批修）
- [ ] 启动时自动加载 + 每 N 步自动落盘（防止崩了丢数据）
- [ ] pytest 保护：重启后 LinUCB 的 arm 拉取次数不归零
- [ ] 教师后台 `/api/lca_debug/<sid>` 端点接通（**v0.56.0 端点已存在，别再虚标**），返回最近 10 条 LCA 决策 + rationale

**为什么排第二**：持久化是 LinUCB 真正"学起来"的前提——不解决这个，v0.56.0 接通了也白搭（每次重启 bandit 从零开始）。

**预估**：1-2 天。

---

### 3.2 v0.58.0 — 双 Agent 互校（CTA 假设 vs LCA 实验验证）

**目标**：把架构文档 §3.3 那 6 步循环真跑起来。

**触发条件**：v0.57.0 完成后 + lbc001 累计答 50+ 题（LinUCB 学到足够 reward 信号）。

**Definition of Done**：
- [ ] 互校协议（`ecos/dual_agent/` 复用 v0.31.0 骨架）：CTA 状态估计**附置信区间**（variance > 阈值 → 触发 LCA 验证）
- [ ] 新增 `LCA.verify_cta_hypothesis(cta_hypothesis, student_id)`：LCA 设计 1-2 道探针题验证 CTA 假设
- [ ] 4 个交互模式至少实现 2 个（**常态 + 冲突**；学习 + 发现排 v0.59.0）
- [ ] 冲突检测：CTA 说"K=0.7" + LCA 实验得 0.4 → CTA 必须修正
- [ ] `/api/judge` 返回值增加 `dual_agent_consensus` 字段（True/False/None）
- [ ] pytest 模拟冲突场景（注入一个"CTA 错估"用例，验证互校能纠错）

**为什么排第三**：互校是 ECOS 核心卖点（"理解+改变"+"抗幻觉"），但**完全依赖前两版**——没有持久化的 LinUCB 和接通的 LCA，互校无从谈起。

**预估**：4-5 天。

**为什么排最后**：H3 是验证项，不是开发项。前面 3 版都完成才有东西可验证。

## 4. 关键决策点（需要 Bisen 拍板）

| 决策 | Mavis 建议 | 备选 |
|------|---------|------|
| **LCA 启动方式** | Feature flag 默认关，v0.56.0 跑 1 周 lbc001 数据后再开 | 直接开（风险高，LinUCB 冷启动可能推烂题） |
| **Reward 信号定义** | `(K_new - K_old) + 0.5 × bloom_layer_progress`（v0.56.0 简化版） | 只用 K 变化（信号弱，学不动）|
| **互校实现深度** | v0.58.0 实现 2/4 模式（常态 + 冲突） | 全 4 模式（工程量翻倍，2 周内做不完） |
| **H3 验证数据来源** | lbc001 实测 + 历史回放混合 | 只用模拟数据（不真实） |
| **LCA 持久化范围** | intervention_history + LinUCB A/b 矩阵 + arm 计数 | 全 LCAEngine 状态（含 attribution / ca_state_machine） |
